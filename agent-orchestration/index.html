Overview

Platform Overview

플랫폼 개요

The AI Agent Orchestration Platform is the common backend engine powering all AI features across MSoftech's products. Rather than building AI logic per-project, every agent — regardless of domain — runs through the same standardized pipeline.

모든 프로젝트의 AI 기능을 구동하는 공통 백엔드 엔진입니다. 도메인에 관계없이 모든 에이전트가 동일한 표준 파이프라인을 통과합니다.

The platform's core challenge is the gap between what LLMs produce (free-form text) and what applications need (structured JSON). The solution is a 3-artifact architecture — Registry, Template, Schema — where each artifact is independently versioned and CI/CD managed, eliminating the need for code deployments when prompts change.

LLM이 생성하는 자유형식 텍스트와 앱이 필요로 하는 구조화된 JSON 사이의 간격을 Registry·Template·Schema 3계층 아티팩트 구조로 해결합니다. 각 파일은 독립적으로 CI/CD 관리되어 프롬프트 변경 시 코드 배포가 불필요합니다.

With Live Broadcast Mode, every pipeline stage is streamed in real time — showing exactly which prompt template was used, how data was injected, what the LLM received, and whether the output passed schema validation. This makes AI behavior fully traceable and auditable.

Live Broadcast Mode를 통해 파이프라인의 모든 단계가 실시간으로 시각화됩니다. AI가 어떻게 판단했는지 완전히 추적·검증할 수 있는 XAI 기술이 접목되어 있습니다.

① Request Intake 요청 접수
The client application (Flutter App) sends the patient data + agent request parameters to the Spring Boot API server via REST API over HTTPS. This includes the data to be analyzed and the target agent identifier.
클라이언트 앱(Flutter)에서 분석할 환자 데이터와 에이전트 요청 정보를 Spring Boot API 서버로 REST API/HTTPS로 전송합니다.
② Prompt Generation 프롬프트 생성
The engine queries the Prompt Registry to retrieve the latest template version for the requested agent. It then loads the Template (.md), collects the incoming request data, and performs Placeholder substitution — injecting live patient data into the CONTEXT section of the template — to produce the final Assembled Prompt.
Registry에서 최신 템플릿을 조회하고, 요청 데이터를 템플릿의 Placeholder에 주입하여 완성된 Assembled Prompt를 생성합니다. 이 Placeholder 치환이 플랫폼의 핵심 기술입니다.
③ LLM Call LLM 호출
The fully assembled prompt is sent to Google Gemini (gemini-2.5-flash-lite) via Google Vertex AI. The platform supports multi-provider routing — the same pipeline can switch between Gemini and OpenAI GPT-4 at runtime without changing agent logic.
완성된 프롬프트를 Google Vertex AI를 통해 Gemini로 전송합니다. 런타임에서 Gemini/OpenAI를 전환할 수 있는 멀티 프로바이더 라우팅을 지원합니다.
④ Schema Validation 응답 검증
The AI response is automatically validated against the predefined Output Schema (JSON Schema draft-07). Validation checks three things: required field enforcement, type validation (string, array, object), and component reuse via $ref / $defs. If validation fails, the output is flagged and logged — it never reaches the client in a broken state.
AI 응답이 Output Schema와 100% 일치하는지 자동 검증합니다. 필수 필드 강제·타입 검증·$ref 컴포넌트 조합 세 가지를 확인하며, 검증 실패 시 재시도/로깅 처리합니다.
⑤ Result Delivery 결과 전송
The validated, schema-compliant final data is mapped to a DTO and sent back to the client application. For healthcare projects, results are also persisted to the FHIR R4 database under the KR Core schema before delivery.
검증된 최종 데이터를 DTO로 변환하여 클라이언트 앱으로 전송합니다. 헬스케어 프로젝트의 경우 KR Core FHIR R4 데이터베이스에도 함께 저장됩니다.
Architecture

3-Layer Artifact Structure

3계층 아티팩트 구조 — Registry · Template · Schema

registry.yaml
Registry
The agent directory. Maps each agent to its Template and Schema files, and manages versioning. A single entry per agent — enabling CI/CD-based prompt management with rollback at any time.
에이전트 디렉토리. 각 에이전트의 템플릿·스키마 파일 경로를 매핑하고 버전을 관리합니다. 프롬프트 변경 시 코드 배포가 불필요합니다.
id name version tags templateFile outputSchemaFile
agent_name.md
Template
The agent's behavioral specification. Written in a strict 5-section structure: MISSION → CONTEXT → RULES → Analysis Guidelines → DATA & FORMAT. The CONTEXT section contains Placeholders that receive live data at runtime.
에이전트의 행동 규범. MISSION→CONTEXT→RULES→분석지침→DATA&FORMAT 5섹션 구조로 작성. CONTEXT의 Placeholder에 실시간 데이터가 주입됩니다.
MISSION CONTEXT RULES 분석지침 DATA & FORMAT
agent_name_schema.json
Output Schema
The contract for AI output structure. Uses JSON Schema draft-07 with $ref / $defs for component-based type reuse. Enforces required fields, type validation, and structural consistency — guaranteeing the AI never returns a broken format.
AI 출력 구조를 강제하는 계약. JSON Schema draft-07 표준, $ref/$defs 컴포넌트 재사용 설계. 환각현상에 의한 불규칙 출력을 구조적으로 방지합니다.
required type $ref $defs draft-07
Prompt Engine

Template 5-Section Structure

모든 템플릿이 따르는 표준 구조 — 에이전트 출력 일관성의 기반

MISSION
목적 정의
Defines the agent's purpose and objective. Sets the scope of what the agent is and is not responsible for analyzing.
CONTEXT
데이터 참조 (Placeholder 주입)
Contains Placeholders that are replaced with live request data at runtime. This is where patient records, measurements, and domain data are injected into the prompt.
RULES
분석 규칙
Hard constraints the agent must follow. Defines what it must always do, what it must never do, and how to handle edge cases and ambiguous data.
Analysis Guidelines
분석 지침 · 판단 기준
Domain-specific judgment criteria. For medical agents, this includes clinical thresholds, risk stratification rules, and interpretation standards.
DATA & FORMAT
출력 구조 정의
Specifies exactly what JSON structure the agent must produce. This section directly mirrors the Output Schema — ensuring the LLM understands the required output format before the schema validation gate.
Why This Structure?

Every agent uses this exact 5-section format — no exceptions. This uniformity is what guarantees consistent output quality across all domains. A diabetes analysis agent and a geological analysis agent have completely different content, but structurally identical templates.

모든 에이전트가 동일한 5섹션 구조를 사용합니다. 도메인이 달라도 구조가 동일하기 때문에 일관된 출력 품질이 보장됩니다.

Placeholder Substitution

The engine reads the template, finds all Placeholder markers in the CONTEXT section, and replaces them with the actual patient/domain data from the incoming request. The result is the Assembled Prompt — a complete, data-rich instruction ready for the LLM.

엔진이 CONTEXT의 Placeholder를 실제 요청 데이터로 치환하여 최종 Assembled Prompt를 완성합니다.

Output Validation

Schema Validation

AI 출력 정확도·일관성을 100% 구조적으로 보장하는 게이트

LLMs produce free-form text by nature — but applications need exact JSON structures. Schema Validation is the gate that automatically closes this gap 100% of the time.
AI는 자유형식 텍스트를 생성하는 본질이지만, 앱에서 사용하려면 정확한 구조의 JSON이어야 합니다. Schema Validation이 이 간격을 100% 자동으로 닫습니다.
🔒
Required Field Enforcement
Every field marked required in the schema must be present in the AI output. Missing fields cause an immediate validation failure — the output is never silently incomplete.
🏷️
Type Validation
Each field's data type is strictly enforced — string, array, object, number, boolean. A field that should be an array cannot arrive as a string without triggering validation failure.
🔗
$ref / $defs Component Reuse
Repeated type structures are defined once in $defs and referenced via $ref. For example, a TimelineItem type shared across blood glucose records, lab results, and medication history.
Observability · XAI

Live Broadcast Mode

파이프라인 전 단계 실시간 시각화 — Explainable AI 구현

When enabled, every stage of the pipeline streams in real time. Each step can be expanded to reveal the actual artifact used — the Registry entry, the injected Input Data, the full Template, the final Assembled Prompt, and the Output Schema used for validation. This makes AI behavior completely transparent and auditable.

활성화하면 파이프라인의 각 단계가 실시간으로 시각화됩니다. 각 단계를 Expand하면 실제 사용된 아티팩트(Registry·Input Data·Template·Assembled Prompt·Output Schema)를 직접 확인할 수 있습니다.

🔄
Post-Deploy Prompt Verification
After deploying a new template, confirm the actual data-merged result in real time before it reaches production users.
🔍
AI Response Quality Audit
Compare the Assembled Prompt side-by-side with the AI response to evaluate output quality and prompt effectiveness.
🐛
Schema Mismatch Debugging
When validation fails, instantly see the exact diff between the Output Schema and the actual AI response to pinpoint the issue.
🚀
New Agent Development
Preview the data-injected prompt result during template authoring — validate the injection logic before the agent goes live.
Deployment

Agent Deployment Map
